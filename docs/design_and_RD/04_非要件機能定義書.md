## **2. 本書の目的**

本書は、Training Bot RAG Hub の品質・安全性・運用性・保守性・コスト制御に関する非機能要件を定義する。

本RAGは、PMTP内のTraining Bot、AI Analysis UI、Bot検証画面へ判断材料を提供する参照基盤であり、売買注文を実行するシステムではない。

したがって、非機能要件では以下を最優先とする。

```text
1. 誤発注を誘発しないこと
2. RAGから注文APIへ到達できないこと
3. LLM出力を無検証で利用しないこと
4. Secret / API Key / JWT / 個人情報をLLMへ送信しないこと
5. すべての重要操作を監査可能にすること
6. コスト上限を超えた場合は安全側に停止・制限すること
7. RAG障害がTrading Engineへ波及しないこと
```

## **4. 安全性要件**

### **4.1 基本方針**

Training Bot RAG Hub は、金融システムとして「便利さ」より「壊れないこと」を優先する。

RAGの役割は、Training BotやAI Analysis UIに対して以下を提供することに限定する。

```text
- 市場文脈
- 過去類似ケース
- 支持材料
- 反対材料
- リスク要因
- Citation
- Confidence
- Provider情報
```

RAGは以下を実行してはならない。

```text
- 注文実行
- 注文API呼び出し
- Bot設定変更
- Bot起動
- Bot停止
- 緊急停止解除
- Risk Limit変更
- API Key参照
- Secret参照
- AI単独発注
```

---

### **4.2 安全性要件一覧**

|**ID**|**要件**|**優先度**|**受け入れ基準**|
|---|---|---|---|
|NFR-SAFE-001|RAGはOrder Serviceの書込APIを呼び出せないこと|Must|RAG ModuleからOrder Service write系clientをimportできない|
|NFR-SAFE-002|RAG回答の`order_permission`は常に`false`であること|Must|Schemaで`z.literal(false)`を強制する|
|NFR-SAFE-003|RAGはBot設定を自動変更できないこと|Must|Bot設定更新APIへの依存が存在しない|
|NFR-SAFE-004|RAGはBot起動・停止・緊急停止解除を実行できないこと|Must|Bot制御APIへの書込権限が存在しない|
|NFR-SAFE-005|RAGの回答は投資助言ではなく検証用参考情報として表現すること|Must|禁止表現検知で断定表現をBLOCKまたは安全化する|
|NFR-SAFE-006|Human-in-the-loopを維持すること|Must|RAG回答から直接注文確定に遷移しない|
|NFR-SAFE-007|RAG障害時にTrading Engineを停止・変更しないこと|Must|RAG障害はRAG API内で閉じる|
|NFR-SAFE-008|二重注文防止設計にRAGを関与させないこと|Must|注文ID生成、Idempotency Key生成をRAGが行わない|

## **6. LLM出力信頼性要件**

### **6.1 基本方針**

LLM出力は、どのProviderを使っても信頼済みデータとして扱わない。  
必ずSchema Validation、Guardrail、Citation検証を通過したものだけを返却する。

---

### **6.2 Output Validation要件**

|**ID**|**要件**|**優先度**|**受け入れ基準**|
|---|---|---|---|
|NFR-LLM-001|RAG回答はJSON Schema / Zodで検証する|Must|Schema不一致時は返却しない|
|NFR-LLM-002|`supporting_factors`を必須にする|Must|空配列なら再生成またはBLOCK|
|NFR-LLM-003|`opposing_factors`を必須にする|Must|空配列なら再生成またはBLOCK|
|NFR-LLM-004|`risk_level`を必須にする|Must|欠落時はBLOCK|
|NFR-LLM-005|`confidence`を0〜1で必須にする|Must|範囲外はBLOCK|
|NFR-LLM-006|`citations`を必須にする|Must|根拠なし回答は返却しない|
|NFR-LLM-007|`order_permission=true`を絶対に許可しない|Must|Schemaでfalse固定|
|NFR-LLM-008|Schema不一致時は最大1回だけ再生成する|Should|再生成失敗時はRAG_SCHEMA_INVALID|
|NFR-LLM-009|Provider raw responseをそのまま返却しない|Must|API Responseは必ず正規化済み|

---

### **6.3 Hallucination抑制要件**

|**ID**|**要件**|**優先度**|**受け入れ基準**|
|---|---|---|---|
|NFR-LLM-010|Citationなしの主張を禁止する|Must|citation未紐付けの重要主張は削除または低信頼扱い|
|NFR-LLM-011|「根拠不十分」の回答を許可する|Must|無理に結論を出さない|
|NFR-LLM-012|予測市場データを事実として扱わない|Must|「市場参加者の見方」として表現する|
|NFR-LLM-013|SNS情報を高信頼ソースとして扱わない|Must|reliability_scoreを低く設定する|
|NFR-LLM-014|過去類似ケースを未来保証として表現しない|Must|「過去ケースであり将来を保証しない」と表現する|

RAGはPMTPの参照系コンポーネントであり、RAG障害が売買基盤へ影響してはならない。

## **27. 最終方針**

Training Bot RAG Hub の非機能要件は、金融システムとして安全側に倒す。

MVPでは、機能を増やすよりも以下を優先する。

```text
- 注文系との完全分離
- LLM出力のSchema固定
- Secret漏洩防止
- Prompt Injection対策
- 監査ログ保存
- コスト上限制御
- Provider Adapterによるロックイン回避
```

RAGはPMTPの売買判断エンジンではない。  
RAGは、Training BotとAI Analysis UIに対して、根拠・反対材料・リスク・引用を提示する参照基盤である。

  

注文可否は、必ず Trading Engine / Risk Filter / Human Confirm 側で判断する。
