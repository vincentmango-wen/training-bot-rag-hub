# 2. システム概要

## 2.1 システム名称

Training Bot RAG Hub

---

## 2.2 システム位置づけ

Training Bot RAG Hubは、Personal Multi Trading Platform内で作成予定のトレーニングボットが参照する知識検索・根拠生成基盤である。

本RAGは、売買注文を直接実行しない。  
役割は、トレーニングボット、AI分析画面、バックテスト機能、将来のBot改善機能に対して、以下の情報を提供することである。

- 市場データの文脈
    
- 過去類似ケース
    
- Bot判断理由
    
- リスク要因
    
- 外部センチメント
    
- ニュース・予測市場情報
    
- 内部ログ・約定履歴・注文履歴
    
- 戦略ドキュメント
    
- 検証レポート
    
- LLM Provider比較評価結果
    

---

## 2.3 システム化背景

PMTPは、株・FX・仮想通貨を統合管理する個人用トレーディングプラットフォームである。

既存設計では、AI分析やBot制御機能が予定されているが、金融システムにおいてAIの出力をそのまま発注に使うことは危険である。

特に以下のリスクが存在する。

- AIハルシネーション
    
- 誤った市場解釈
    
- 外部情報のノイズ
    
- SNS情報の偏り
    
- 予測市場データの過信
    
- Bot暴走
    
- 誤発注
    
- 二重注文
    
- 説明不能な売買判断
    
- 判断根拠の監査不能
    
- LLM Provider依存
    
- LLM API障害
    
- LLM出力形式の不安定化
    
- LLMコスト増加
    
- 外部APIへの機密情報送信リスク
    

そのため、初期段階ではRAGを「注文判断エンジン」ではなく、「トレーニングボットが参照する知識・根拠・検証支援レイヤー」として導入する。

また、LLM ProviderはOpenAI APIをMVP第一候補とするが、OpenAIに固定しない。  
Provider Adapterを導入し、Anthropic Claude、Google Gemini、Mistral AI、Local LLMへ差し替え可能な構成とする。

# 3. 基本方針

## 3.1 設計方針

|方針|内容|
|---|---|
|Read-only First|RAGは参照専用とする|
|No Direct Trading|RAGから注文APIを直接呼び出さない|
|Human-in-the-loop|最終判断は人間または既存ルール側で行う|
|Explainability|根拠・理由・反対材料を必ず提示する|
|Source-grounded|回答には参照ソースを紐づける|
|Risk-aware|常にリスク評価を含める|
|Auditability|問い合わせ・参照・回答を保存する|
|Fail Safe|異常時は回答停止または低信頼として扱う|
|OpenAI First|MVPではOpenAI APIを第一候補にする|
|Vendor Lock-in Never|OpenAI固定実装は禁止する|
|Provider Adapter Mandatory|LLM Provider差し替え用Adapterを必須とする|
|Model Evaluation Driven|Provider選定は評価結果に基づいて見直す|

---

## 3.2 最重要原則

```text
RAGは注文しない。
RAGはBotに判断材料を渡すだけ。
注文可否はTrading Engine / Risk Filter / Human Confirm側で判断する。
LLM ProviderはOpenAIを第一候補とするが、固定しない。
```

---

## 3.3 既存PMTPとの関係

Training Bot RAG Hubは、PMTPの以下機能と連携する。

|PMTP機能|RAGとの関係|
|---|---|
|Market Service|市場データをRAG検索対象にする|
|AI Analysis Service|RAG要約・根拠をAI分析画面に表示する|
|Bot Service|Training BotからRAGを参照する|
|Portfolio Service|過去損益・ポジション情報を分析対象にする|
|Order Service|注文履歴を参照する。ただし注文実行はしない|
|Audit Log|RAG問い合わせ・回答履歴を保存する|
|Notification Service|高リスク検知時に通知連携する|
|LLM Provider Adapter|OpenAI / Claude / Gemini / Mistral / Local LLMを抽象化する|

```text
    ↓

    ↓
Reranking
    ↓

    ↓
LLM Answer Generation
    ↓
Output Validation
    ↓

    ↓

    ↓

```

---

### 4.2.2 Bot判断説明フロー

```text

    ↓

    ↓

    ↓

    ↓

    ↓
```

### 4.2.3 類似ケース検索フロー

```text

    ↓
```

# 5. スコープ

## 5.1 対象範囲

|分類|対象|
|---|---|
|内部データ|注文履歴、約定履歴、ポジション履歴、資産履歴、Botログ、AI分析履歴、監査ログ|
|市場データ|OHLCV、Tick、板情報、出来高、RSI、MACD、ATR、VWAP、Funding Rate、Open Interest|
|ドキュメント|Bot戦略ルール、リスク設定、PMTP設計書、運用ルール|
|外部データ|ニュース、予測市場データ、SNS要約、マクロイベント情報|
|RAG機能|検索、要約、類似ケース抽出、理由生成、リスク抽出、引用提示|
|Bot連携|Training Bot向けRead-only API|
|UI連携|AI分析画面、Bot検証画面、RAG回答履歴画面|
|監査|RAG問い合わせ、検索結果、回答、利用Bot、参照ソースの保存|
|LLM Provider|OpenAI、Anthropic Claude、Google Gemini、Mistral AI、Local LLMへの拡張設計|
|Embedding Provider|OpenAI、Gemini、Voyage AI、Mistral、Local Embeddingへの拡張設計|

---

## 5.2 対象外範囲

|対象外|理由|
|---|---|
|RAGによる注文実行|誤発注リスクが高い|
|RAGによるBot自動起動|Bot暴走リスクがある|
|RAGによるBot設定自動変更|リスク制御を破る可能性がある|
|RAGによる緊急停止解除|安全設計に反する|
|利益保証・勝率保証|不可能|
|投資助言としての断定表現|法務・倫理リスクが高い|
|Polymarket等での取引実行|法規制・利用規約リスクがある|
|高頻度取引用リアルタイム判断|RAGは低遅延売買に不向き|
|OpenAI専用実装|Provider Lock-inになるため禁止|
|LLM出力の無検証利用|金融システムとして危険|

# 7. ユースケース要件

## UC-RAG-001 市場文脈検索

|項目|内容|
|---|---|
|概要|現在の相場に関連する内部・外部情報を検索する|
|主アクター|Training Bot / 一般ユーザー|
|入力|symbol、timeframe、query、対象期間|
|出力|市場要約、関連要因、リスク、参照ソース|
|優先度|最優先|

### 基本フロー

1. Training BotがRAG Query APIへ問い合わせる
    
2. Queryを検証する
    
3. symbol、timeframe、期間で検索対象を絞る
    
4. 内部データ・外部データを検索する
    
5. 関連度順に再ランキングする
    
6. LLM Provider Adapter経由で回答生成する
    
7. 回答Schemaを検証する
    
8. 回答を保存する
    
9. Training Botへ返却する
    

---

## UC-RAG-002 Bot判断理由生成

|項目|内容|
|---|---|
|概要|Botが出した仮シグナルに対して説明を生成する|
|主アクター|Training Bot|
|入力|bot_signal、features、market_context|
|出力|支持材料、反対材料、リスク、confidence|
|優先度|最優先|

### 重要ルール

- BUY / SELL / HOLDを断定的な投資指示として表現しない
    
- 「参考情報」「検証用」「根拠候補」として返却する
    
- `order_permission` は常に `false` とする
    
- LLM出力はJSON Schemaで検証する
    
- Schema不一致時は再生成またはBLOCKとする

# 9. LLM Provider要件

## 9.1 LLM Provider基本方針

MVPではOpenAI APIを第一候補とする。  
ただし、OpenAI APIに固定しない。

LLM Providerは必ずAdapter経由で利用し、以下Providerへ差し替え可能にする。

- OpenAI
    
- Anthropic Claude
    
- Google Gemini
    
- Mistral AI
    
- Local LLM
    
- 将来追加Provider
    

---

## 9.2 OpenAI APIをMVP第一候補とする理由

|観点|理由|
|---|---|
|Structured Outputs|RAG回答をJSON Schemaへ固定しやすい|
|Embedding|Embedding APIが同一Providerで利用できる|
|TypeScript連携|Zod / JSON Schemaとの親和性が高い|
|実装速度|MVPで最も実装しやすい|
|運用単純性|生成とEmbeddingを同一Providerで始められる|
|Schema Validation|固定レスポンス形式が必要な金融RAGに向いている|

---

## 9.3 OpenAI APIの制約

|制約|内容|対策|
|---|---|---|
|Provider依存|OpenAI障害時に影響を受ける|Fallback Providerを用意する|
|外部API送信|データを外部に送る必要がある|Secret除外、Masking、送信範囲制御|
|コスト増加|Token利用量に比例して費用が増える|モデル階層化、Cache、要約前処理|
|モデル仕様変更|APIやモデル仕様変更の影響を受ける|Provider Adapterで吸収する|
|金融専用モデルではない|金融ドメイン特化ではない|内部評価セットで精度評価する|

---

## 9.4 Provider比較要件

|Provider|強み|弱み|PMTPでの扱い|
|---|---|---|---|
|OpenAI|Structured Outputs、Embedding、SDK、実装速度のバランスが良い|外部API依存、金融専用ではない|MVP第一候補|
|Anthropic Claude|長文理解、慎重な説明、リスクレビューに強い|Embeddingは別Providerが必要|リスクレビュー・反対材料生成の候補|
|Google Gemini|Structured Output、Embedding、Web Grounding系との相性が良い|Google系基盤への依存が増える|外部情報RAGの候補|
|Mistral AI|Structured Output、Open-weight活用、自前運用余地|日本語・金融RAG評価が必要|将来のコスト削減・自前化候補|
|Local LLM|外部送信不要、データ保護に強い|GPU・運用・精度評価が必要|本番高度化フェーズ候補|

## 18.2 採用理由

MVPでは、専用Vector DBを追加するよりもPostgreSQL + pgvectorを優先する。

理由：

- 既存PMTP構成と相性がよい
    
- 運用対象が増えにくい
    
- MVPとして十分
    
- メタデータとEmbeddingを同一DBで扱いやすい
    
- 将来Qdrant、Weaviate、OpenSearch等へ移行可能
    

LLMはOpenAI APIをMVP第一候補とする。

理由：

- Structured OutputsによりSchema固定しやすい
    
- Embeddingと生成を同一Providerで開始できる
    
- TypeScript / Zod連携しやすい
    
- MVP実装速度が高い
    
- 運用開始時の複雑性を抑えられる
    

ただし、OpenAI固定は禁止する。  
Provider Adapterにより、他Providerへ差し替え可能にする。

- Bot設定変更
    
- 緊急停止解除
    
- API Key参照

- 利益保証
    
- 勝率保証
    
- 外部テキストの命令実行

- LLM出力の無検証利用
    
- OpenAI専用実装

# 25. 最終要件方針

Training Bot RAG Hubは、PMTP内のトレーニングボットが参照する知識基盤として構築する。

初期MVPでは、RAGは以下に限定する。

- 内部データ検索
    
- Bot判断理由生成
    
- 類似ケース検索
    
- リスク要因抽出
    
- 根拠付き回答
    
- 監査ログ保存
    
- Training BotへのRead-only連携
    
- OpenAI APIによるMVP実装
    
- Provider Adapterによる将来差し替え対応
    

一方で、以下は明確に禁止する。

- RAGによる注文実行
    
- RAGによるBot設定変更
    
- RAGによる自動売買判断
    
- RAGによる投資助言の断定
    
- RAGによるSecret参照
    
- OpenAI専用の密結合実装
    
- LLM出力の無検証利用
    

LLM Providerについては、以下を正式方針とする。

```text
OpenAI First
Vendor Lock-in Never
Provider Adapter Mandatory
Evaluate Before Expansion
```

この方針により、AI・RAGをPMTPへ安全に組み込みながら、金融システムとして最重要である注文安全性、資産保護、監査可能性を維持する。
