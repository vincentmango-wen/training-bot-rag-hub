# **2. アーキテクチャ設計方針**

## **2.1 最重要原則**

```text
RAGは注文しない。
RAGはBotに判断材料を渡すだけ。
注文可否は Trading Engine / Risk Filter / Human Confirm 側で判断する。
```

Training Bot RAG Hubは、PMTPにおける知識参照・根拠生成・類似ケース検索・リスク説明のための補助基盤である。

RAGは以下を行わない。

- 注文実行
- Bot起動
- Bot停止
- Bot設定変更
- 緊急停止解除
- Risk Limit変更
- 出金処理
- 投資助言の断定
- 利益保証
- 勝率保証

# **11. Provider選択アーキテクチャ**

## **11.1 Provider Policy**

|**Task Type**|**Primary**|**Fallback**|**方針**|
|---|---|---|---|
|normal_summary|OpenAI mini系|Gemini / Mistral|低コスト優先|
|bot_signal_explanation|OpenAI|Claude / Gemini|構造化出力優先|
|risk_review|Claude|OpenAI|慎重なリスク説明|
|external_news_summary|Gemini / OpenAI|Mistral|外部情報要約|
|embedding|OpenAI small|Gemini / Mistral|MVPはOpenAI|
|confidential_analysis|Local LLM|none|外部送信禁止|

---

## **11.2 Fallback方針**

Fallback条件：

- Provider API障害
- Timeout
- Rate Limit
- Schema Validation失敗
- Safety Violation
- Cost Limit超過

Fallback後も失敗した場合：

```text
LLM回答は返さず、検索結果のみ返す
```

# **17. 障害時アーキテクチャ**

## **17.1 障害パターンと挙動**

|**障害**|**挙動**|
|---|---|
|LLM Provider障害|Fallback Providerへ切替|
|全Provider障害|検索結果のみ返却|
|Embedding生成失敗|Retry後DLQ|
|外部データ取得失敗|内部データのみで回答|
|pgvector検索失敗|エラー返却、Trading Engine影響なし|
|Guardrail BLOCK|回答を返さずBLOCK理由を返却|
|Schema Validation失敗|再生成、失敗時BLOCK|

## **17.2 Fail Safe方針**

```text
RAGが失敗しても注文系は止めない。
ただし、RAG結果を必要とするBot検証処理は低信頼または停止扱いにする。
```

# **21. 最終アーキテクチャ方針**

Training Bot RAG Hubは、PMTPの注文系とは分離された参照専用RAG基盤として構築する。

MVPでは、PostgreSQL + pgvector + Redis + NestJS + OpenAI APIを採用し、ローカルDocker Composeで開始する。

ただし、OpenAI固定にはしない。  
LLM Provider AdapterとEmbedding Provider Adapterを必須とし、Claude、Gemini、Mistral、Local LLMへ将来切り替え可能な構造にする。

本アーキテクチャの結論は以下である。

```text
1. RAGはTrading Engineと直接接続しない
2. RAGはRead-only参照基盤として実装する
3. RAG回答は必ずSchema Validationする
4. order_permissionは常にfalse
5. Provider AdapterによりOpenAI固定を避ける
6. pgvectorでMVPを開始する
7. Provider利用量・コスト・Latencyを必ず保存する
8. 障害時はFail Safeで検索結果のみ返す
9. 外部データはPhase 2以降に段階導入する
10. クラウド常時稼働はMVP後に判断する
```
