# **2. 目的**

Training Bot RAG Hubは金融システム配下のRAG基盤である。

そのため単純なデータ保存ではなく、

- 検索精度
- 回答品質
- Explainability
- Provider比較評価
- 監査性
- 再現性

を維持するためのデータ品質管理が必要となる。

本設計書は以下を目的とする。

```text
1. 品質の悪いデータをRAGへ流入させない
2. データ品質を定量評価する
3. 品質劣化を検知する
4. 回答品質低下を未然防止する
5. 監査可能な状態を維持する
```

企画書で定義された「Source-grounded」「Explainability」「Auditability」を実現するための設計である。

# **3. 品質管理方針**

## **3.1 基本方針**

|**方針**|**内容**|
|---|---|
|Garbage In Garbage Out防止|低品質データを隔離する|
|Source First|ソース品質を評価する|
|Quality Score化|品質を数値管理する|
|監査可能性|品質判定履歴を保存する|
|自動化優先|可能な限り自動判定する|
|Fail Safe|品質不明データは低信頼扱い|

# **5. データ品質モデル**

## **5.1 品質スコア**

各データに対して総合品質スコアを付与する。

```text
quality_score =

(
 completeness * 0.25 +
 accuracy * 0.25 +
 consistency * 0.15 +
 freshness * 0.20 +
 reliability * 0.15
)
```

評価範囲

```text
0.0 ～ 1.0
```

---

## **5.2 品質ランク**

|**Score**|**ランク**|**扱い**|
|---|---|---|
|0.90〜1.00|A|優先利用|
|0.75〜0.89|B|通常利用|
|0.60〜0.74|C|低信頼表示|
|0.40〜0.59|D|検索対象外候補|
|0.00〜0.39|E|隔離|

# **6. 品質評価項目**

## **6.1 Completeness（完全性）**

欠損の有無を評価する。

### **市場データ**

必須項目

```text
symbol
timestamp
open
high
low
close
volume
```

欠損率

```text
missing_rate =
missing_fields / required_fields
```

---

### **ニュース**

必須項目

```text
title
published_at
source
content
url
```

---

### **Botログ**

必須項目

```text
bot_id
event_type
timestamp
message
```

---

## **6.2 Accuracy（正確性）**

異常値検知を行う。

### **市場データ**

例

```text
high < low
volume < 0
close = null
```

検知時

```text
quality_status = INVALID
```

---

### **Funding Rate**

閾値

```text
abs(rate) > 10%
```

異常値候補

---

### **Open Interest**

```text
negative value
```

禁止

---

## **6.3 Consistency（一貫性）**

同一データの整合性を確認する。

例

```text
OHLCVと約定履歴が矛盾しない
```

```text
ニュース時刻 > 取得時刻
```

は禁止。

---

## **6.4 Freshness（鮮度）**

要件定義のrecency_scoreを具体化する。  

計算式

```text
freshness_score =
exp(-age_hours / threshold)
```

例

|**データ種別**|**閾値**|
|---|---|
|ニュース|24時間|
|市場データ|1時間|
|SNS|12時間|
|戦略文書|180日|

---

## **6.5 Reliability（信頼性）**

要件定義のreliability_scoreを実装する。

# **17. 最終方針**

Training Bot RAG Hubでは、

```text
検索品質 = データ品質
回答品質 = データ品質 × Provider品質
```

として扱う。

そのため品質管理は単なるETL監視ではなく、

- データ品質
- ソース品質
- RAG品質
- Provider品質

を統合管理する。

特に金融システムとして重要な

```text
誤発注誘発ゼロ
Prompt Injection汚染ゼロ
Secret漏洩ゼロ
Hallucination最小化
```

を達成するため、品質スコアリング・隔離・監査ログを標準機能として実装する。企画書および要件定義で定義された「Explainability」「Auditability」「Risk-aware」を品質管理面から担保する。
